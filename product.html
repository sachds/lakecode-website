How Lakecode works

Governed workflows, persistent memory, and structured evidence — from your terminal to your entire organization.

The CLI experience

22 built-in workflows for real Databricks and Snowflake engineering tasks. Install in 30 seconds, running workflows in under two minutes.

1

Install and connect

npm install -g lakecode — sign in with Google OAuth, connect your Databricks or Snowflake workspace. Lakecode automatically profiles your catalogs, schemas, jobs, and clusters.

2

Run workflows or chat

Use slash commands for structured workflows (/debug, /prove, /cost) or describe what you need in plain English. Every action is classified before execution — reads run immediately, writes require your approval.

3

Get structured evidence, not chat transcripts

Every workflow produces shareable artifacts — debug reports, evidence packs, cost breakdowns — with provenance. Attach them to PRs, incidents, Jira tickets, or audit trails.

Debugging & Monitoring

/debug job <id> Full diagnosis: config, failures, logs, queries, root-cause summary with historical context
/status job <id> Job config + last 5 runs at a glance
/logs <run-id> Run output + AI error analysis
/queries job <id> SQL query history for the latest run

Data Quality

/prove <table> Evidence pack: schema, row counts, nulls, freshness, stats, duplicates, samples + health assessment
/audit jobs Scan for SLA gaps, missing timeouts, recurring failures — risk-scored report

Cost & Platform Ops

/cost top Top spend drivers from system billing tables
/cost spike Compare day vs baseline; AI identifies likely drivers
/drift detect Compare deployed resources against bundle definitions
/uc explain-access Trace why a principal has (or lacks) access to a Unity Catalog object

Deployment

/deploy <local> <ws-path> Import a local file into the workspace
/run job <id> Trigger job run + poll until completion
/capture job <id> Capture a job or pipeline into a Databricks Asset Bundle

Safety classification

Every action is classified before execution. No surprises against production.

READ — runs immediately

  • Fetching job configs, run history, logs
  • Querying billing tables, cluster metrics
  • Profiling schemas, checking access patterns

WRITE — requires approval

  • Deploying files to workspace
  • Triggering job runs
  • Modifying configurations

DESTRUCTIVE — explicit confirmation

  • Dropping tables or schemas
  • Deleting jobs or clusters
  • Revoking access grants

Configurable policy engine

  • Define which operations require approval per org
  • Set default catalogs, schemas, naming conventions
  • Enforce guardrails that match your security posture

Engram: persistent institutional memory

The knowledge layer that makes every session smarter than the last.

What it stores

Dependencies, ownership, incident history, cost patterns, healthy baselines, resolution playbooks, operational SLAs. Structured as claims with provenance — not raw text embeddings.

How it learns

Every workflow execution contributes observations. Debug a job? The failure pattern and resolution get recorded. Prove a table? The baseline stats become the reference. The knowledge graph grows organically with use.

Role-scoped access

Different teams see what's relevant to them. The data engineering team sees pipeline health; the platform team sees cost trends and cluster utilization. Scoped by role, not by chat history.

Claims, not embeddings

Engram stores structured claims — "job 4827 failed due to data skew on 2026-04-10, resolved by salting join key" — with provenance, not 500K tokens of raw context. 200 tokens of structured claims replace half a million tokens of re-discovery.

Survives attrition

When someone leaves, their operational expertise stays. The knowledge graph is persistent and organizational, not personal. New hires inherit the full context on day one.

Governed and auditable

Every claim has a source, a timestamp, and a confidence level. You can inspect what the agent knows, correct it, and trace how it informed a decision. No black box.
