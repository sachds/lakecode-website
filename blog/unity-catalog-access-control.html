Managing Unity Catalog Permissions from the Terminal

March 28, 2026 · Workflow Guide

Unity Catalog permissions are powerful but hard to reason about. Grants cascade through catalogs, schemas, and tables. Groups inherit from other groups. And when something goes wrong — "why can't this service principal read this table?" — the debugging process usually involves clicking through multiple Databricks UI screens.

Lakecode's /uc commands bring this entire workflow to the terminal.

Tracing access: /uc explain-access

The most common question: "Does this principal have access to this object, and if so, how?"

lakecode /uc explain-access data-eng@company.com analytics.prod.orders

This walks the entire grant hierarchy from the catalog level down to the specific table. It shows:

  • Direct grants on the object
  • Inherited grants from parent schemas and catalogs
  • Group memberships that contribute permissions
  • The effective permission set (SELECT, MODIFY, etc.)

You can also test for a specific action:

lakecode /uc explain-access data-eng@company.com analytics.prod.orders --action SELECT

This returns a clear yes/no answer with the full explanation chain.

Diffing grants: /uc diff-grants

For teams managing permissions as code, you often have a desired-state YAML that describes who should have access to what. The /uc diff-grants command compares reality against that desired state:

lakecode /uc diff-grants analytics.prod --to grants.yml

The output is a reviewable plan showing:

  • Additions — grants that exist in the desired state but not in UC
  • Removals — grants that exist in UC but not in the desired state
  • No change — grants that already match

Think of it like terraform plan for Unity Catalog permissions. You review the diff before applying anything.

Applying changes: /uc apply-grants

Once you've reviewed the plan, apply it:

lakecode /uc apply-grants --plan abc123

This executes the grant/revoke statements with full safety measures:

  • Takes a before-snapshot of current grants
  • Applies each change individually
  • Takes an after-snapshot for comparison
  • Produces an audit report showing exactly what changed

The before/after snapshots serve as an audit trail. If something goes wrong, you can see exactly what was changed and when.

A typical workflow

  1. A new team member needs access to the analytics catalog
  2. Update your grants.yml to add their group
  3. Run /uc diff-grants to see what will change
  4. Review the plan in your terminal
  5. Run /uc apply-grants to execute
  6. The audit report is saved for compliance

The entire process takes under a minute and produces a paper trail. No clicking through the UI, no guessing about what permissions cascade where.
