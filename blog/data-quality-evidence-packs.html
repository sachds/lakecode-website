Building Data Quality Evidence Packs with /prove

March 28, 2026 · Workflow Guide

"Is this table healthy?" is a question data engineers answer dozens of times a week. Usually it involves writing ad-hoc SQL: count the rows, check for nulls, look at the latest timestamp, spot duplicates. The /prove command automates all of this into a single evidence pack.

Running /prove

lakecode /prove analytics.prod.orders

This kicks off a structured workflow that runs multiple checks against the table and produces a comprehensive health report. It works on both Databricks and Snowflake.

What the evidence pack contains

Schema snapshot

Full column listing with data types, nullable flags, and any comments. This is your baseline for understanding the table structure.

Row count & freshness

Total row count plus the most recent value of any timestamp column (e.g., updated_at, _loaded_at). If the freshest record is older than expected, you'll see a warning.

Null analysis

For each column: count and percentage of null values. Flags columns where nulls are unexpectedly high (e.g., a supposedly required field has 30% nulls).

Duplicate detection

Checks for duplicate rows on likely key columns. If the table has a clear primary key pattern, it tests that. Otherwise, it checks for full-row duplicates.

Numeric statistics

For numeric columns: min, max, mean, median, and standard deviation. Helps spot outliers and data drift (e.g., a price column suddenly has negative values).

Sample rows

A small sample of recent rows for visual inspection. Often the quickest way to spot obvious issues.

Health assessment

An overall assessment that synthesizes all the checks into a pass/warn/fail verdict with specific findings. This is the summary you'd paste into a Slack thread or Jira ticket.

Output format

Results are saved to the ./evidence/ directory as a structured report. The AI also provides a conversational summary in the terminal. You can reference the evidence pack in code reviews, incident reports, or compliance documentation.

Common use cases

  • Post-deployment validation — Run /prove on key tables after a pipeline deployment to verify data integrity.
  • Incident investigation — When a downstream dashboard looks wrong, prove the source table to find where the data quality broke down.
  • Periodic health checks — Run /prove on critical tables daily or weekly as part of your data observability practice.
  • Onboarding — New team members can quickly understand a table's shape, quality, and freshness without writing any SQL.

Try it

npm install -g lakecode
lakecode login
lakecode /prove <catalog>.<schema>.<table>

A typical evidence pack takes 10-20 seconds and costs $0.10-$0.30 depending on the number of columns and checks involved.
