Debugging Failed Databricks Jobs with a Single Command

March 28, 2026 · Workflow Guide

A Databricks job fails at 3 AM. By the time you see the alert, you need to figure out: what task failed, what error was thrown, what queries ran, whether the cluster had issues, and whether this is a data problem or a code problem. That usually means opening 4-5 different tabs and cross-referencing information manually.

The /debug job command does all of this in one shot.

What happens when you run /debug job

lakecode /debug job 123456

Behind the scenes, Lakecode executes a structured workflow with multiple steps. Here's exactly what it does:

1. Fetch the job configuration

Pulls the full job definition from the Databricks Jobs API — schedule, tasks, cluster config, parameters, timeout settings. This gives context about what the job is supposed to do.

2. Find the most recent failed run

Lists recent runs and identifies the latest failure. Extracts the run ID, start time, duration, and which specific task(s) failed.

3. Pull task-level error details

For each failed task, fetches the full error output: Python tracebacks, SQL errors, notebook cell outputs. This is the raw evidence.

4. Check query history

Queries the SQL Statement History API to find all queries executed during the failed run. Identifies slow queries, failed queries, and any that returned unexpected results.

5. Inspect cluster events

Pulls cluster events (startup, autoscaling, termination) to check if the failure was infrastructure-related — OOM kills, spot instance preemption, driver crashes.

6. Produce a root cause analysis

With all the evidence collected, the AI analyzes the full picture and produces a structured diagnosis: what failed, why, and what to do about it. If the fix is straightforward (e.g., a typo in SQL, a missing permission), it can suggest a specific code change.

What the output looks like

The final output is a structured report that includes:

  • Summary — one-line root cause
  • Failed task — which task in the multi-task job failed and at what stage
  • Error evidence — the specific error message and traceback
  • Contributing factors — cluster issues, data problems, or upstream dependencies
  • Recommended fix — actionable steps to resolve the issue

When to use /debug vs other commands

If you already know the run ID and just want logs, use /logs <run-id> — it's faster and cheaper. If you want to check a job's schedule and recent run history without deep analysis, use /status job <id>.

/debug job is for when something is broken and you don't know why. It collects everything and connects the dots.

Snowflake equivalent

On Snowflake, the equivalent command is /debug task <name>. It follows the same pattern: fetch the task definition, pull run history, extract errors, and produce a diagnosis — adapted for Snowflake's task execution model.

lakecode /debug task DAILY_REFRESH

Try it

Install Lakecode and point it at a failed job:

npm install -g lakecode
lakecode login
lakecode /debug job <your-job-id>

The whole workflow typically takes 15-30 seconds and costs $0.30-$0.80 in credits depending on the complexity of the failure.
