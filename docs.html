Getting Started

Install

Lakecode is distributed via PyPI. Install it in any Python 3.9+ environment:

pip install lakecode

This installs the lakecode CLI globally. Verify it works:

lakecode --version

Login

Authenticate with your Google account. This grants Lakecode access to manage your credits and proxy AI requests on your behalf.

lakecode login

This opens your default browser for Google OAuth sign-in. Once authorized, a license key is saved locally and used for all subsequent requests.

If you are on a headless server or SSH session where a browser is not available, use the device flow instead:

lakecode login --device

This prints a URL and a code. Open the URL on any device, enter the code, and sign in with Google. The CLI will detect the authorization and save your key automatically.

First command

Run your first evidence pack against any table in your Databricks workspace:

lakecode /prove my_catalog.my_schema.my_table

This generates a full health assessment: schema snapshot, row counts, null analysis, freshness, duplicates, numeric stats, and sample rows.

Workspace profiling

On first run, Lakecode automatically profiles your Databricks workspace. It discovers catalogs, schemas, tables, and warehouses available to your account. This profile is cached locally so future commands have instant context about your environment. You do not need to configure anything — it happens automatically.

Authentication

Google OAuth

Lakecode uses Google OAuth as the primary sign-in method. There are two flows:

  • Web flow (default) — lakecode login opens a browser tab. Best for local development machines.
  • Device flow — lakecode login --device prints a URL + code. Best for headless servers, SSH sessions, or CI environments.

Both flows result in a license key being saved to your local machine. You stay signed in until the key expires or you explicitly sign out.

License keys

After OAuth, Lakecode issues a license key in the format lk_<base64url>. The key is an Ed25519-signed payload containing your email, plan type, and expiry date. It is stored locally and sent with every request to the proxy, which verifies the signature server-side.

You can view your current key on the dashboard or by running:

lakecode /status

Bring Your Own Key (BYOK)

If you prefer to use your own AI provider keys and bypass the Lakecode proxy entirely, set one of these environment variables:

# Use your own Anthropic key
export ANTHROPIC_API_KEY=sk-ant-...

# Or use your own OpenAI key
export OPENAI_API_KEY=sk-...

When a BYOK key is set, Lakecode routes requests directly to the provider. No credits are consumed, no proxy is involved, and billing is handled entirely by your provider account.

Commands Reference

All commands are invoked as slash commands within the Lakecode CLI. Each command connects to your Databricks workspace using your existing CLI profile and auth.

Observability

/status job <id>

Retrieves the job configuration and its last 5 runs. Use this as a quick health check for any job — see the schedule, cluster config, and recent run outcomes at a glance.

lakecode /status job 123456
/logs <run-id>

Fetches the full output of a specific run, including stdout, stderr, and error tracebacks. Lakecode analyzes the errors and highlights the root cause.

lakecode /logs 78901234
/debug job <id>

Full diagnosis workflow: pulls the job config, finds the most recent failed run, extracts errors, checks query history, and produces a root cause analysis. This is the single command for “why is this job broken?”

lakecode /debug job 123456
/queries job <id>

Retrieves the SQL query history for the latest run of a job. Useful for understanding what queries a job executed and how long each took.

lakecode /queries job 123456
/queries run <run-id>

Retrieves the SQL query history for a specific run. Use this when you need to inspect queries from a particular execution, not just the latest.

lakecode /queries run 78901234

Data Quality

/prove <table>

Generates a comprehensive evidence pack for a table: schema snapshot, row count, null analysis, freshness check, duplicate detection, numeric statistics, sample rows, and an overall health assessment. Results are saved to the ./evidence/ directory.

lakecode /prove my_catalog.my_schema.my_table
/audit jobs

Scans all jobs in your workspace for SLA gaps, missing timeout configurations, and recurring failures. Produces a risk-scored report so you can prioritize which jobs need attention.

lakecode /audit jobs

Cost Analysis

/cost top

Queries your Databricks billing tables and identifies the top spend drivers — which jobs, clusters, or warehouses are costing the most.

lakecode /cost top
/cost spike

Compares today’s spend against a baseline period to detect cost spikes. Identifies the specific drivers behind any increase.

lakecode /cost spike

Deployment & Execution

/deploy <local-path> <workspace-path>

Imports a local file (notebook, SQL, Python) into your Databricks workspace at the specified path.

lakecode /deploy ./notebooks/etl.py /Workspace/production/etl
/run job <id>

Triggers a job run and polls for completion. Shows real-time status updates until the run finishes or fails.

lakecode /run job 123456

Configuration

Conventions files

Lakecode reads conventions files to understand your team’s standards and preferences. There are two levels:

  • Global — ~/.lakecode/conventions.md applies to all projects on your machine.
  • Project — .lakecode/conventions.md in your project root applies only to that project. Project conventions override global ones.

Conventions are written in plain Markdown. Lakecode reads them before every command to tailor its behavior. Example:

# Team Conventions

## Default catalog and schema
Always use `analytics.prod` as the default catalog.schema unless specified.

## Naming patterns
- Tables: snake_case, prefixed with source (e.g., stripe_charges)
- Jobs: kebab-case with team prefix (e.g., data-eng-daily-refresh)

## Preferred warehouse
Use the `analytics-medium` SQL warehouse for ad-hoc queries.

## Standards
- All tables must have a `_loaded_at` timestamp column
- Partition by date for tables over 1B rows
- Tag all jobs with team name and SLA tier

Skills injection

Lakecode ships with a library of Databricks-specific skills (Unity Catalog, Delta Lake, workflows, SQL warehouses, etc.). When you run a command, Lakecode detects relevant keywords and injects the appropriate skills into the AI context. This means the AI always has deep, current knowledge about the Databricks features involved in your query — without you configuring anything.

Environment variables

Lakecode uses the standard Databricks CLI environment variables for workspace connectivity:

  • DATABRICKS_HOST — Your workspace URL (e.g., https://dbc-abc123.cloud.databricks.com)
  • DATABRICKS_TOKEN — A personal access token for your workspace
  • ANTHROPIC_API_KEY — (Optional) Your own Anthropic key for BYOK mode
  • OPENAI_API_KEY — (Optional) Your own OpenAI key for BYOK mode

If you have a Databricks CLI profile configured (~/.databrickscfg), Lakecode will use it automatically. Environment variables take precedence over the config file.

Billing & Credits

Free tier

Every account receives free daily credits. Credits reset at midnight UTC. No credit card is required. This is enough for a few commands per day, depending on complexity.

Pro plan

For heavier usage, add a payment method on your dashboard. Once a card is on file, auto-reload kicks in:

  • Default reload amount is $49 (adjustable to any amount $5+)
  • When your balance drops below 20% of your reload amount, your card is charged automatically
  • You can pause, adjust, or manually trigger reloads from the dashboard
  • Free daily credits still apply — they are used first before paid credits

Cost model

Each request is billed based on actual AI tokens consumed (input + output). Costs vary by command complexity:

  • Simple commands (/status, /logs) — typically $0.05–$0.10
  • Medium commands (/prove, /queries) — typically $0.10–$0.30
  • Complex workflows (/debug, /audit) — typically $0.30–$0.80

Exact costs depend on the amount of data the AI needs to process (number of tables, query history length, error verbosity, etc.).

BYOK and billing

If you set ANTHROPIC_API_KEY or OPENAI_API_KEY, requests go directly to the provider. No Lakecode credits are consumed and no proxy markup applies. You are billed directly by Anthropic or OpenAI at their standard rates.

Dashboard

Manage everything from your dashboard: current balance, reload settings, charge history, and license key. The dashboard updates in real time as you use commands.

Evidence Packs

What are evidence packs?

Evidence packs are structured, machine-readable outputs from Lakecode workflows. They capture the data quality metrics and diagnostics that the AI gathered during execution, formatted as standalone artifacts you can share, attach, or archive.

The /prove evidence pack

When you run /prove on a table, Lakecode generates an evidence pack containing:

  • Schema snapshot — Column names, types, and nullability as of the check time
  • Row count — Total rows in the table
  • Null analysis — Per-column null counts and percentages
  • Freshness check — When the table was last updated, how stale it is
  • Duplicate detection — Checks for duplicate rows on key columns
  • Numeric statistics — Min, max, mean, stddev for numeric columns
  • Sample rows — A small sample of actual data for visual inspection
  • Health assessment — An overall pass/warn/fail verdict with explanations

Output location

Evidence packs are saved to the ./evidence/ directory in your current working directory. Each pack is timestamped and named after the table.

Use cases

  • Pull requests — Attach evidence packs to PRs that modify table schemas or ETL logic as proof that downstream data is healthy.
  • Incident response — Attach to incident tickets to document the state of affected tables at the time of investigation.
  • Audit trails — Keep a history of data quality checks for compliance and governance.
  • Stakeholder communication — Share structured metrics with non-technical stakeholders instead of raw query output.

Security & Safety

Local execution

Lakecode runs entirely on your machine. It connects to Databricks using your existing CLI profile and credentials. Your Databricks token never leaves your machine — it is used locally to make API calls directly to your workspace.

Permissions

Lakecode respects Unity Catalog permissions. If your Databricks account cannot access a catalog, schema, or table, neither can Lakecode. There is no privilege escalation.

Plan mode

For complex or potentially risky operations, Lakecode enters plan mode. Before executing, it shows you exactly what it intends to do and asks for confirmation. This prevents accidental modifications to production resources.

Guardrails

Built-in guardrails prevent common mistakes: dropping tables without confirmation, running unbounded queries, or deploying to production paths without explicit intent. These guardrails are always active and cannot be disabled.

Proxy security

  • License keys are Ed25519 signed and verified server-side on every request
  • The proxy does not log request or response content — only metadata (user ID, token counts, timestamps) for billing
  • Payments are handled by Stripe (PCI DSS compliant) — Lakecode never sees your card number
  • Revoked keys are checked against a KV store on every request for instant invalidation

Architecture

System overview

Lakecode consists of three components that work together:

  • CLI — The Python package installed on your machine. Handles Databricks API calls, workspace profiling, skills injection, and user interaction.
  • Proxy Worker — A Cloudflare Worker that sits between the CLI and Anthropic’s API. Validates license keys, tracks token usage, deducts credits, and forwards requests.
  • Auth Worker — A Cloudflare Worker that handles Google OAuth flows, license key generation, Stripe payment processing, and billing webhooks.

Request flow

CLI (your machine)
 |
 |-- Databricks API calls (direct, using your token)
 |
 |-- AI requests --> Proxy Worker (Cloudflare)
                      |-- Verify license key (Ed25519)
                      |-- Check balance / deduct credits
                      |-- Forward to Anthropic API
                      |-- Return response to CLI

Data storage

  • D1 (SQLite) — User records, balances, charge history, and reload settings
  • KV — Key revocation lists and reload flags for fast edge lookups
  • Local cache — Workspace profiles (catalogs, schemas, tables, warehouses) are cached on your machine for fast context loading

License key structure

A license key (lk_<base64url>) is an Ed25519-signed payload containing:

  • User email
  • Plan type (free or pro)
  • Expiry timestamp
  • Issuer signature

The signing key is held by the Auth Worker. The corresponding public key is embedded in the Proxy Worker for verification. Keys cannot be forged or tampered with.

Workspace profiling

On first run (or when the cache is stale), Lakecode calls the Databricks REST API to discover your workspace topology: catalogs, schemas, tables, and SQL warehouses. This profile is cached locally in ~/.lakecode/ and refreshed periodically. It allows the AI to understand your environment without you having to describe it every time.
